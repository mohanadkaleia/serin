# ENG-53 — M0: Repo scaffolding, tooling, and CI bootstrap

**Milestone:** M0 — Protocol spike
**Tech-lead:** planning complete; implementation delegated per assignments below.
**TDD refs:** §1.1 (repo layout), §4.1 (stack), §2 (event model — informs `core/` placement), §13 (M0 exit criterion).

## Goal (restated)

Stand up the empty-but-real Python 3.12 project skeleton so that every later M0 ticket
(ENG-54 envelope, ENG-55 JCS, ENG-56 hashing) lands into a working, linted, typed, tested,
CI-gated repo. Nothing in this ticket implements protocol logic — it builds the container that
protocol logic drops into. The one load-bearing architectural requirement is that the shared
event library, `msgd.core`, is importable from **both** server and CLI code, and that `msgctl`
is a real installable console script.

Areas touched: repo root (tooling), `server/` (package + tests), `cli/` (package + tests), CI.
No `web/`, no Docker, no compose in this ticket.

## Packaging decision — uv **workspace** with two members (`server` → `msgd`, `cli` → `msgctl`)

I choose a `uv` workspace whose root `pyproject.toml` is a non-package (virtual) coordinator holding
the shared dev tooling config, with two buildable members: `server/` (package `msgd`, containing the
`core/` subpackage) and `cli/` (package `msgctl`, declaring a workspace dependency on `msgd`). This is
the simplest shape that satisfies the TDD §1.1 tree *as written* — `server/msgd/core/` and `cli/` are
siblings, not one flat package — while making `import msgd.core` work from CLI code through a real
dependency edge (a `uv` workspace source, resolved editable, no path hacks) and giving `msgctl` its own
`[project.scripts]` entry point. It also anticipates M1 cleanly: the server member will grow heavy deps
(FastAPI, asyncpg, SQLAlchemy, Alembic) that the workspace keeps scoped to `msgd`, and because `msgctl`
ships inside the server image anyway (TDD §11), the CLI depending on the whole `msgd` package is
operationally free. A single flat project was rejected because it would force server-only runtime deps
into the CLI's declared dependency set and blur the §1.1 boundary the whole project navigates by; a
separately published `msgd-core` third member was rejected as premature (see Open Questions).

## Implementation Plan

### Target file tree (all created by this ticket)

```text
msg/
  pyproject.toml            # workspace root: members + shared ruff/mypy/pytest config + dev group
  uv.lock                   # generated by `uv sync` (committed)
  .python-version           # "3.12" — pins the interpreter uv provisions
  .gitignore                # .venv, __pycache__, *.egg-info, .pytest_cache, .mypy_cache, .ruff_cache
  server/
    pyproject.toml          # member package "msgd" (hatchling)
    msgd/
      __init__.py           # __version__ = "0.1.0"
      core/
        __init__.py         # empty-but-real; docstring naming the ENG-54/55/56 modules to come
    tests/
      test_smoke.py         # placeholder test + `import msgd.core` proof (server side)
  cli/
    pyproject.toml          # member package "msgctl"; depends on msgd (workspace); console script
    msgctl/
      __init__.py           # __version__ = "0.1.0"
      cli.py                # argparse main(); `msgctl --version`; imports msgd.core to prove the edge
    tests/
      test_cli.py           # `msgctl --version` behavior + `import msgd.core` proof (cli side)
  .github/
    workflows/
      ci.yml                # ruff check, ruff format --check, mypy, pytest on py3.12 via uv
```

### File-by-file purpose and content outline

**`/pyproject.toml`** (workspace root — virtual, not built)
- `[project]`: `name = "msg-workspace"`, `version = "0"`, `requires-python = ">=3.12"`,
  `dependencies = ["msgd", "msgctl"]` (so `uv sync` installs both members editable into one venv).
- `[tool.uv.workspace]`: `members = ["server", "cli"]`.
- `[tool.uv.sources]`: `msgd = { workspace = true }`, `msgctl = { workspace = true }`.
- `[tool.uv]`: `package = false` (root is a coordinator, nothing to build/install for it).
- `[dependency-groups]` `dev = ["pytest>=8", "ruff>=0.6", "mypy>=1.11"]`.
- `[tool.ruff]`: `target-version = "py312"`, `line-length = 100`; `[tool.ruff.lint]`
  `select = ["E","F","I","UP","B"]`; ruff formatter used (so `ruff format --check` is meaningful).
- `[tool.mypy]`: `python_version = "3.12"`, `strict = true`,
  `files = ["server/msgd", "server/tests", "cli/msgctl", "cli/tests"]`. (Editable workspace install
  means mypy resolves `msgd` from the venv; no `mypy_path` gymnastics needed.)
- `[tool.pytest.ini_options]`: `testpaths = ["server/tests", "cli/tests"]`, `addopts = "-q"`.

**`/server/pyproject.toml`** (member `msgd`)
- `[project]`: `name = "msgd"`, `version = "0.1.0"`, `requires-python = ">=3.12"`, `dependencies = []`
  (FastAPI/SQLAlchemy/etc. arrive in M1, not here).
- `[build-system]`: `requires = ["hatchling"]`, `build-backend = "hatchling.build"`.
- `[tool.hatch.build.targets.wheel]`: `packages = ["msgd"]`.

**`/server/msgd/__init__.py`** — `__version__ = "0.1.0"`. Nothing else.

**`/server/msgd/core/__init__.py`** — empty-but-real. Module docstring stating this is the shared
event library consumed by server + CLI, and that `envelope`, `jcs`, `hashing`, `schemas` (and
`testdata/vectors.json`) land here in ENG-54/55/56. **Constraint noted in the docstring:** `core`
must stay free of server-only deps (FastAPI/SQLAlchemy) so the CLI can import it cheaply.

**`/server/tests/test_smoke.py`** — (a) a trivial `test_placeholder` assertion so pytest has a
collectible test; (b) `test_core_importable_from_server` doing `import msgd.core` and asserting the
module object exists. Proves core importability from the server side (acceptance criterion).

**`/cli/pyproject.toml`** (member `msgctl`)
- `[project]`: `name = "msgctl"`, `version = "0.1.0"`, `requires-python = ">=3.12"`,
  `dependencies = ["msgd"]`.
- `[project.scripts]`: `msgctl = "msgctl.cli:main"`.
- `[tool.uv.sources]`: `msgd = { workspace = true }`.
- `[build-system]`: hatchling; `[tool.hatch.build.targets.wheel]` `packages = ["msgctl"]`.

**`/cli/msgctl/__init__.py`** — `__version__ = "0.1.0"`.

**`/cli/msgctl/cli.py`** — `main(argv: list[str] | None = None) -> int` using `argparse` with a
`--version` action printing `msgctl {__version__}`. Imports `msgd` (or `msgd.core`) at module top so
the shared-library dependency edge is exercised at runtime, not just in tests. Fully type-annotated
(mypy strict).

**`/cli/tests/test_cli.py`** — (a) `test_version` invoking `main(["--version"])` (capture SystemExit/
stdout) and asserting the version string; (b) `test_core_importable_from_cli` doing `import msgctl`
then `import msgd.core` — proves `msgd.core` is importable from CLI-side code (acceptance criterion).

**`/.github/workflows/ci.yml`** — outline:
- `name: CI`; `on: [push, pull_request]`.
- One job `checks`, `runs-on: ubuntu-latest`.
- Steps:
  1. `actions/checkout@v4`.
  2. `astral-sh/setup-uv@v4` (official, pinned major) with uv caching enabled.
  3. `uv python install 3.12` (or rely on `.python-version`).
  4. `uv sync --dev` (installs both members editable + dev group).
  5. `uv run ruff check`.
  6. `uv run ruff format --check`.
  7. `uv run mypy`.
  8. `uv run pytest`.
- Keep it single-job, single-Python, no matrix (TDD/devops principle: fast, boring). Forward note:
  the M0 **rebuild ≡ incremental** equivalence check becomes a permanent required step in a later
  M0 ticket (ENG-56/57 area) — leave the pytest step as the natural home for it; do not add it now.

can author `ci.yml` in parallel against this spec and merge once the package skeleton exists.

### Test plan

- `server/tests/test_smoke.py::test_placeholder` — pytest has something to collect (satisfies "trivial
  placeholder test so pytest runs").
- `server/tests/test_smoke.py::test_core_importable_from_server` — `import msgd.core` succeeds.
- `cli/tests/test_cli.py::test_version` — `msgctl --version` prints the expected version and exits 0.
- `cli/tests/test_cli.py::test_core_importable_from_cli` — `import msgd.core` succeeds from CLI-side code
  (this is the load-bearing acceptance proof that the shared library crosses the package boundary).
- Tooling gates (all must pass locally and in CI): `uv run ruff check`, `uv run ruff format --check`,
  `uv run mypy`, `uv run pytest`.
- Manual smoke: `uv run msgctl --version` from a clean `uv sync`.

### Risks / open questions

- **uv workspace install semantics.** `uv sync` installs the root's dependencies; because the virtual
  root declares `dependencies = ["msgd","msgctl"]` with workspace sources, both members install editable.
  If a uv version behaves differently, the fallback is `uv sync --all-packages`. python-engineer should
  confirm both members are importable in the synced venv and adjust the root deps vs. `--all-packages`
  flag accordingly, noting the final choice here.
- **CLI pulls server deps (M1 onward).** `msgctl` depends on the whole `msgd` package, so when M1 adds
  FastAPI/asyncpg to `msgd`, the CLI's environment inherits them. Accepted for MVP: `msgctl` ships inside
  the app image (TDD §11), so there is no separate lightweight CLI distribution to protect. **Open
  question deferred, not decided now:** if a standalone/lightweight `msgctl` ever matters, extract
  `core/` into a third workspace member `msgd-core` that both `msgd` and `msgctl` depend on. Keeping
  `core` free of server-only imports (enforced by the docstring constraint and, later, an import-linter
  rule) preserves that option at zero cost. Do **not** do it in this ticket.
- **Build backend choice.** Plan assumes `hatchling` (uv's default, zero-config for the `packages=[...]`
  src layout used here). No blocker; noted so the two members stay consistent.
- **`.python-version` vs CI.** Ensure the pinned `3.12` in `.python-version` matches the Python the CI
  provisions, so local and CI resolve identically.
